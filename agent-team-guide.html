需求变更流程

根据变更发起者不同，变更会沿着不同的协作链传播，确保所有环节同步更新

场景 1：完整需求变更（产品需求分析师发起）

适用情况：产品功能调整、业务逻辑变更、需要全流程更新的重大需求变更

场景 2：原型变更（原型设计师发起）

适用情况：UI 样式调整、交互优化、页面布局变更等不涉及后端和数据库的前端变更

场景 3：数据库变更（数据库建模师发起）

适用情况：数据表新增/修改、索引优化、数据迁移等数据库层面的变更

场景 4：后端代码变更（后端工程师发起）

适用情况：API 优化、Bug 修复、性能调优等仅涉及后端代码的变更

知识持续同步机制

让上下文文档、代码、数据库结构、API 契约、测试用例和运行配置在持续开发中保持一致

设计理念：原生优先，刻意不用胶水层

需求文档是「快照」，代码会随迭代漂移。三个月后 module 文档写的还是旧逻辑，却没有任何机制提示「这份文档已和代码冲突」—— 这就是需求与代码偏移的根源。我们的解法不依赖任何第三方目录约定（如 .harness/ 这类需要靠指针跳转、可能被绕过的胶水层），而是把规则写进 Claude Code / Codex 会自动加载的原生入口文件，让规则「每次都在场」。

✓ 保留的两个内核

① 知识带「来源状态」（current / needs-verification / conflict）
② 每次改动收尾强制回写

✗ 砍掉的复杂度

不引入 .harness/ 目录、不要 sync-status.json、不搭向量库 / 图数据库 —— 只有两个工具，不为不存在的需求付税

每次工作的固定动作（可持续闭环）

任何 Agent 接到任务都按此顺序走，收尾的「知识同步闸」把成果回写，下次才查得到 —— 形成自我维持的循环。

三层落地结构

① 控制层入口

CLAUDE.md（Claude Code 自动读）
AGENTS.md（Codex 自动读）
两份内容一致、手动同步。含固定动作、Agent 路由表、知识地图、风险分级、技术栈速查。

② 知识同步闸

8 个 Agent 定义末尾各加「收尾：知识同步闸」。
Claude 侧在 .claude/agents/*.md；Codex 侧统一收敛到 .codex/skills/<name>/，由 SKILL.md、agents/openai.yaml、references/legacy-role.md 承接。

③ 当前变更留痕

docs/CHANGELOG.md —— 同步闸的回写目标。
只记录当前有效的 durable 变更：日期 ｜ 角色 ｜ 知识类型 ｜ 摘要 ｜ 影响文档 ｜ 来源状态。

知识同步闸：收尾时逐条自问

每次有意义的变更收尾时，判断本次是否动了以下任一类 durable 知识：

架构 / 模块边界 数据 / API 契约 功能行为 / 算法阈值 模块归属 复发故障模式 durable 决策 / 工作规则

✅ 动了

更新对应文档（见下方映射表）+ 在 CHANGELOG.md 追加一行。发现代码与文档冲突时，来源状态标 conflict 并通知责任角色，不沉默放过。

⛔ 没动

一句话说明「本次为 XX 类改动，未触及 durable 知识，无需回写」。纯文案 / 纯局部重构不必记录。

公共一致性契约：任何 Agent 改需求、页面、前端、后端、数据库、测试或部署配置时，都必须同步检查上下游契约；暂时不能同步的缺口必须写明风险、责任角色和补齐位置。

📂 文档分类速记（写之前对号入座）

受 llmwiki「概念 / 决策 / 操作 / 参考 / 模板」分类启发， docs/ 在原有 requirement/ · api/ · database/ · fixes/ · deploy/ 之外，新增 5 个分类目录。 旧文档不强制重构：未来再写"为什么 / 取舍 / 操作"的内容时优先放新目录， 老 PRD 里这类内容在被实际查阅时再外迁。

内容性质判定提问去哪里
要做什么功能"用户/业务想要什么"docs/requirement/
为什么这么设计（不变的原则）"这套系统的世界观"docs/concepts/
在 A/B/C 中选 B（有取舍）"为什么不是另一种做法"docs/decisions/（ADR）
怎么执行（可重复操作）"新人按这个跑就能 work"docs/runbooks/
外部链接 / 论文 / 官方文档"会被多次引用的外部源"docs/references/
反复使用的样板"下次还要照这个写一份"docs/templates/
实际代码 / SQL / 接口"实现层"backend/、frontend/、docs/api/、docs/database/

每个新目录的 README.md 都写了模板与规则，写之前点进去看一眼即可： concepts/README.md、decisions/README.md、runbooks/README.md、templates/README.md。

在 Claude Code CLI 中激活 Agent

Claude Code CLI 通过 Skill 机制激活 Agent，使用斜杠命令即可快速启动对应角色。

# 方式 1：直接使用斜杠命令 /市场需求分析师 分析在线教育市场 # 方式 2：使用英文命令 /market-analyst 分析在线教育市场 # 查看所有可用 Agent /help

说明：Agent 配置文件位于 .claude/agents/ 目录，每个 Agent 对应一个 skill 配置文件。

在 Codex CLI 中激活 Agent

Codex 侧真正可触发的是 .codex/skills/<name>/SKILL.md 这套 skill。使用时可显式写 $skill-name，也可在自然语言里点名角色；项目根目录的 AGENTS.md 会作为总则自动加载。

# 启动 Codex CLI（交互模式） codex # 方式 1：显式使用 Codex skill $product-manager 基于 docs/requirement 重新整理这个功能 PRD # 方式 2：自然语言点名角色 frontend-engineer 检查这个页面的表单交互和 API 接入是否完整，并补齐缺失部分 # 方式 3：在 CLI 里查看或选择技能 /skills $

说明：Codex 当前唯一使用层是 .codex/skills/。其中 SKILL.md 是精简入口，agents/openai.yaml 承接显示元信息，完整角色说明通过 references/legacy-role.md 承接；不再维护 .codex/agents/*.toml。

使用示例

# Claude Code：启动市场分析 /market-analyst 分析在线教育市场的用户需求和竞品情况 # Codex：生成产品需求文档 $product-manager 基于现有代码和 docs/requirement，补齐「策略项目」模块 PRD # Codex：设计交互原型 $prototype-designer 根据 PRD 设计课程列表页和详情页原型 # Codex：前端开发 $frontend-engineer 根据 PRD 检查 frontend/web 里对应页面，补齐真实 API、loading/error/success 状态 # Codex：后端实现 $backend-engineer 根据前端调用和 docs/api，实现缺失接口并补测试 # Codex：质量验证 $qa-engineer 对刚才改动做回归测试，输出未覆盖风险

最佳实践

  • 按照流程顺序启动 Agent，确保上游交付物完成后再启动下游
  • 在 Codex 中优先使用 $skill-name 或直接点名角色；角色入口统一维护在 .codex/skills/<name>/SKILL.md
  • 前端工程师在原型设计师产出的 React 组件上做数据接入与架构补齐，不重做 UI
  • 需求变更时，先通过产品需求分析师评估影响范围
  • 每个阶段完成后，使用测试工程师进行质量验证
  • 部署前确保所有测试用例通过，运维工程师负责最终上线
  • 保持文档同步更新，确保团队信息一致性
