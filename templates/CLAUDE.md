## 一、每次工作的固定动作（最重要）

任何 Agent 接到任务，**按这个顺序**走，不要跳：

1. **先查知识** —— 动手前先读与任务相关的已有文档（见下方「知识地图」），确认"现在的真相是什么、代码在哪、有没有已知问题"。**当前代码永远优先于过时文档**：文档与代码冲突时，以代码为准，并记下冲突。
2. **再验证代码** —— 用搜索/读文件确认现状，不要凭文档或记忆假设。
3. **执行变更** —— 小步、单一变量、复用已有组件与流程。
4. **收尾过「知识同步闸」** —— 见第三节。这是堵住"需求与代码偏移"的关键，**不可跳过**。

## 三、知识同步闸（收尾强制项）

### 公共一致性契约

所有 skills / agents team 工作时，都必须维护**上下文文档、代码、数据库结构、API 契约、测试用例、部署/运行配置**之间的一致性。任何变更不能只停留在单一层：

- 改需求或页面契约 → 同步检查原型、前端实现、API/数据模型和测试影响。
- 改前端字段、交互或状态 → 同步检查页面需求文档、API 入参出参、mock 数据和测试用例。
- 改后端接口、服务逻辑或错误码 → 同步检查 API 文档、前端调用、数据库字段/索引和回归测试。
- 改数据库 schema、索引或迁移 → 同步检查 ORM/Schema、服务查询、初始化数据、数据字典和测试夹具。
- 改测试或部署配置 → 同步检查实际代码路径、运行环境变量、部署文档和故障恢复说明。
- 修复某一端（Web / Desktop / Extension 等）的共享资源、依赖或行为 bug → 同步检查其他端是否存在同源问题；同一根因的 bug 必须一次性覆盖所有端，不能"一端修、另一端留"。

**两类知识，写法不同**：

- **项目内的真相**（架构/契约/行为）→ 写进 `docs/` 下对应文档 + `docs/CHANGELOG.md`。
- **跨会话的偏好/约定/踩坑**（下次会话也该记得）→ 写进 `~/.claude/projects/.../memory/`（一事一文件 + MEMORY.md 索引）。
- **用户给出的反馈或纠正** → 先判断是否构成 durable 工作规则（"下次会话/其他 Agent 也应遵守"）。是的话，**立刻**更新 CLAUDE.md + AGENTS.md 对应章节，而不是只在当次任务应用。

每次有意义的变更**收尾时**，逐条自问 —— 这次是否动了：

- **架构** —— 系统结构、模块边界、技术选型
- **数据/API 契约** —— 表结构、字段、接口入参出参、协议
- **功能行为** —— 用户可见行为、业务逻辑、算法/阈值/公式
- **模块归属** —— 某块知识由哪个文件/服务负责
- **复发故障模式** —— 踩过且会再踩的坑
- **durable 决策 / Agent 工作规则** —— 长期有效的设计选择或协作约定

**判定与动作**：

- ✅ **动了** → 更新对应文档，并在 `docs/CHANGELOG.md` 追加一行。
- ⛔ **没动** → 一句话说明"本次为 XX 类改动，未触及 durable 知识，无需回写"。

## 五、风险分级

- **文案/字符串改动**：直接做。
- **小改动**（局部、低风险、不动共享契约）：直接做，收尾过同步闸。
- **大改动**（动共享路径/契约/schema/架构/Agent 行为）：先给计划 + 影响范围 + 测试点，**经用户确认再做**。
- **难回退/对外动作**（删多文件、改生产配置、数据迁移、推远端）：先说清风险与是否可逆，**等用户明确同意**。

## 七、反模式（别做）

- 把本文件堆成巨型 prompt —— 它是总则与指针，细节在各 Agent 定义里。
- 凭过时文档或记忆动手而不验证代码。
- 改了 durable 知识却不回写、也不在 CHANGELOG 留痕。
- 一次性迁移/重写全部文档；每个任务都新造一套抽象或流程。
- 删旧文档不留原始证据。
