# runbooks/ — 操作手册

> **写什么**：可重复执行的操作步骤——本地起服务、生产发版、故障排查、数据迁移、密钥轮转……
> **不写什么**：设计理念（去 `concepts/`）、决策原因（去 `decisions/`）。

## 当前内容

- [local-dev-setup.md](local-dev-setup.md) — 本地开发环境一键启动

## 何时写 runbook

只要符合下面任一条，就值得写一篇：

- 至少有 2 个人会执行的操作。
- 出过事或者会出事的操作（备份、迁移、轮密钥）。
- 新人 onboarding 必经的路径。

## 写作格式

```markdown
---
runbook: <kebab-case 主题>
title: <一句话标题>
owner: <负责角色>
audience: <读者>
last-verified: YYYY-MM-DD   # ⚠️ 最近一次真的跑通的日期
related:
  - <相关 decisions / concepts 路径>
---

# <标题>

> 目的：让 <读者> 在 <预期时间> 内完成 <预期成果>。

## 前置依赖
| 工具 | 版本 | 用途 |

## 步骤 1 — <动作>
代码块 + 期望输出 + 失败时的快速兜底

## 步骤 2 — ...

## 常见故障
| 现象 | 排查 |

## 关停与清理
（如果适用）

## 相关材料
```

## 关键规则

1. **`last-verified` 必填**——超过 90 天没核对的 runbook 标记为"待复核"。
2. **每个步骤都给可复制的命令**——不要写"打开 IDE，点击……"这种无法自动化的描述。
3. **失败兜底放在故障表里**，而不是步骤里——避免主流程被嗯掉。
4. **改了部署/启动相关代码，必须同步对应 runbook**——这是知识同步闸的一部分。
